名词解释

MAC地址：在数据链路层，每一个网卡出厂就自带，作用是在同一局域网内找设备，相当于设备的身份证号

IP地址：在网络层，作用为在网络中定位设备

DNS（域名系统）：作用是把域名与IP地址互相翻译

网关：作用是跨网段访问的出口

子网掩码：作用是判断两个IP在不在同一个局域网内

ARP协议：作用是通过IP找MAC

DHCP：作用是自动分配IP，子网掩码，网关，DNS

HTTP：超文本传输协议，明文传输，不加密，不安全，数据能被捕获

HTTPS：HTTP+加密层（SSL/TLS)，加密传输，安全

TCP：传输层协议，可靠，面向连接，慢一点但稳定，会三次握手四次挥手，用于网页，文件传输，邮件

UDP：传输层协议，不可靠，无连接，快，不握手，发完不管，用于游戏，直播，视频通话，DNS

**SSH：**安全远程登录协议，加密，安全，用于远程连服务器

ICMP：网络层协议，作用是网络诊断，报错

FTP：文件传输协议，作用是传文件

网络模型

OSI七层模型

OSI模型将网络通信过程分为七个层次。每一层都负责特定的网络功能，并为上层提供服务。七层模型包括：

物理层：

物理层主要负责传输原始比特流（0和1）。它涉及网络设备（如电缆、光纤、无线信号等）和硬件接口。

例子：网卡、集线器、物理接口、光纤、电缆等。

数据链路层：

数据链路层的主要作用是提供可靠的物理传输。它将物理层传输的原始数据帧封装成数据帧，并处理错误检测和修正。

例子：以太网（Ethernet）、帧中继（Frame Relay）、MAC地址。

网络层：

网络层的作用是负责数据包的路由和转发。它将数据包从源节点传送到目标节点，并根据目标IP地址决定最佳路径。

例子：IP协议、路由器、ICMP协议、IP地址。

传输层：

传输层负责端到端的可靠通信。它提供数据传输的错误检测和流量控制，并保证数据传输的可靠性（如TCP协议）或不可靠性（如UDP协议）。

例子：TCP（传输控制协议）、UDP（用户数据报协议）。

会话层：

会话层负责建立、管理和终止应用程序之间的会话。它保持会话状态，确保数据在两端的正确传输。

例子：远程登录协议（Telnet）、NetBIOS、RPC（远程过程调用）。

表示层：

表示层负责数据的编码、解码、压缩和加密。它确保不同系统之间的数据格式兼容。

例子：加密算法（如SSL/TLS）、数据压缩、字符编码（如ASCII，UTF-8）。

应用层：

应用层提供直接面向用户的网络服务，如电子邮件、文件传输和Web浏览。

例子：HTTP（超文本传输协议）、FTP（文件传输协议）、SMTP（简单邮件传输协议）、DNS（域名系统）。

TCP/IP四层模型

网络接口层：网卡、以太网、WiFi 等

网络层：IP、ARP、ICMP

传输层：TCP、UDP

应用层：HTTP、HTTPS、FTP、DNS、SSH 等

常见协议与层次的映射

**IP协议：**在网络层工作，负责数据包的寻址和路由。

**TCP协议：**在传输层工作，提供可靠的数据传输服务。

**UDP协议：**在传输层工作，提供不可靠的数据传输服务。

**HTTP/HTTPS协议：**在应用层工作，负责Web内容的请求与响应。

**DNS协议：**在应用层工作，用于域名解析。

**ARP协议：**在数据链路层工作，负责IP地址和MAC地址之间的映射。

网络模型为何分层

  1. 复杂问题简单化
  2. 每层只干自己的事，互不干扰
  3. 标准化，大家都能兼容
  4. 易维护、易排错
  5. 易扩展、易升级

TCP编程流程

TCP（传输控制协议）是一种面向连接的可靠流式服务，意味着在通信前需要建立连接，并且在连接过程中能够确保数据的完整性和顺序。以下是 TCP 编程的基本流程，分别描述了 TCP服务器 和 TCP客户端 如何通过编程进行数据传输。

TCP服务器

结构体

创建套接字(socket())

创建一个套接字用于监听和处理传入的连接请求。

套接字是计算机网络里，让两个程序能互相收发数据的 “接口 / 通道”，是网络通信的基础。在Linux/Unix系统中，Socket本质是一个文件描述符，表现为int类型的整数。

通常使用 socket(AF_INET, SOCK_STREAM, 0) 来创建一个TCP套接字。

成功返回套接字的文件描述符，失败返回-1。

AF_INET 表示IPv4地址族，SOCK_STREAM 表示使用TCP协议。

绑定端口和IP地址(bind())

使用bind()函数将服务器的套接字绑定到一个本地地址(IP)和端口上

bind()将sockfd与一个socket地址绑定，成功返回0，失败返回-1

函数原型

参数：

sockfd：int类型，是套接字描述符，通常通过socket()函数返回

addr：const struct sockaddr*类型,是一个指向sockaddr结构体的指针，包含要绑定的本地地址（即本地IP地址和端口号）

对于IPV4，通常使用struct sockaddr_in

对于IPV6，通常使用struct sockaddr_in6

addrlen:socklen_t类型，是addr结构体的大小，指定了addr指针所指向的地址结构的长度。该参数告诉bind()函数结构体的大小，避免访问错误内存

监听连接请求(listen())

使用 listen() 函数使套接字进入监听状态，准备接受客户端连接请求。

监听时会指定最大连接数（比如 listen(sock, 5)，表示最多同时处理5个连接请求）。

参数说明：

sockfd：已创建但未连接的套接字文件描述符。

backlog：指定挂起连接队列的最大长度。

返回值：

成功时返回 0。

出错时返回 -1，并设置 errno 以指示错误原因。

接受连接(accept())

使用 accept() 函数阻塞并等待客户端的连接。一旦客户端连接到服务器，accept() 会返回一个新的套接字，该套接字专门用于与该客户端的通信。

accept() 返回一个新的套接字和客户端的地址信息。

参数说明：

sockfd：监听套接字描述符（由 socket() 创建，并经过 bind() 和 listen()）。

addr：指向 struct sockaddr 结构的指针，用于存放客户端的协议地址。如果不关心客户端地址，可设置为 NULL。

addrlen：指向 socklen_t 类型变量的指针。调用前应将其初始化为 addr 缓冲区的大小；调用后，内核会将其更新为实际地址长度。

返回值：

成功时返回一个新的套接字描述符（已连接套接字），用于与客户端进行数据通信。

出错时返回 -1，并设置 errno 以指示错误原因。

接受数据(recv())

使用 recv() 函数接收从客户端发来的数据。接收的数据通常是字节流，服务器需要根据协议进行解析。

例如：recv(client_sock, buffer, sizeof(buffer), 0)。

参数说明：

sockfd：已连接的套接字描述符（由 accept() 返回或由 connect() 建立的套接字）。

buf：指向接收数据缓冲区的指针，用于存放接收到的数据。

len：缓冲区的大小，即最多接收的字节数。

flags：接收标志，控制接收行为。常用值包括：

0：默认行为，与 read() 类似。

MSG_PEEK：查看数据但不从缓冲区移除（数据仍可供后续读取）。

MSG_WAITALL：等待直到接收到全部 len 字节的数据（除非发生错误或连接关闭）。

发送数据(send())

使用 send() 函数将数据发送到客户端。

例如：send(client_sock, data, strlen(data), 0)。

参数说明：

sockfd：已连接的套接字描述符（由 accept() 返回或由 connect() 建立的套接字）。

buf：指向要发送数据的缓冲区的指针。

len：要发送的数据长度（字节数）。

flags：发送标志，控制发送行为。常用值包括：

0：默认行为，与 write() 类似。

MSG_DONTROUTE：绕过路由表查找（通常用于本地网络）。

MSG_OOB：发送带外数据（Out-of-Band data）。

关闭连接(close())

数据传输完成后，使用 close() 关闭套接字，断开与客户端的连接。

TCP客户端

创建套接字(socket())

客户端需要先创建一个套接字。与服务器端相似，通常使用 socket(AF_INET, SOCK_STREAM, 0) 来创建一个TCP套接字。

连接到服务器(connect())

使用 connect() 函数向指定的服务器发起连接请求。

connect() 会阻塞直到连接成功或发生错误。

需要提供服务器的IP地址和端口号。

参数说明：

sockfd：客户端套接字描述符（由 socket() 创建）。

addr：指向服务器协议地址结构的指针（例如 struct sockaddr_in 或 struct sockaddr_in6），包含服务器的 IP 地址和端口号。

addrlen：服务器地址结构的大小（字节数）。

返回值：

成功时返回0

出错时返回-1，并设置errno以指示错误原因

发送数据(send())

连接建立后，客户端可以通过 send() 函数将数据发送到服务器。

例如：send(sock, data, strlen(data), 0)。

接受数据(recv())

客户端可以通过 recv() 函数接收来自服务器的数据。

例如：recv(sock, buffer, sizeof(buffer), 0)。

关闭连接(close())

数据传输完成后，使用 close() 关闭套接字，断开与服务器的连接。

代码示例

TCP连接建立：三次握手

在TCP协议中，建立连接的过程称为“三次握手”，其过程如下：

  1. SYN：客户端发送一个SYN（同步）包到服务器，表示客户端请求建立连接。
  2. SYN-ACK：服务器接收到SYN包后，回应一个SYN-ACK包，表示服务器同意建立连接。
  3. ACK：客户端接收到SYN-ACK包后，发送ACK包，表示连接建立成功。

通过三次握手，客户端和服务器之间会确认双方都准备好进行数据传输。

TCP连接断开：四次挥手

当通信完成时，TCP连接需要断开，过程称为“四次挥手”：

  1. FIN：主动关闭连接的一方（如客户端）发送一个FIN包，表示希望关闭连接。
  2. ACK：接收方（如服务器）接收到FIN包后，回复ACK包，表示已接收到关闭请求。
  3. FIN：接收方（如服务器）准备关闭连接时，发送一个FIN包，表示也希望关闭连接。
  4. ACK：主动关闭连接的一方（如客户端）接收到FIN包后，发送ACK包，表示连接已关闭。

UDP编程流程

与 TCP 不同，UDP（用户数据报协议）是一种无连接、不可靠的协议，不需要像 TCP 那样进行连接建立和断开。因此，它在处理数据时更为轻量级，适用于实时通信和对速度要求较高的场景（如视频流、在线游戏等）。

UDP服务器

创建套接字(socket())

服务器使用 socket() 创建一个UDP套接字，通常使用 AF_INET（IPv4）和 SOCK_DGRAM（数据报）参数来创建一个UDP套接字。

例如：socket(AF_INET, SOCK_DGRAM, 0)。

绑定端口和IP地址(bind())

使用 bind() 函数将服务器的套接字绑定到一个特定的端口和IP地址。与TCP不同，UDP服务器不需要调用 listen() 或 accept()。

例如：bind(sock, (struct sockaddr *)&server_addr, sizeof(server_addr))。

接收数据(recvfrom())

服务器使用 recvfrom() 函数接收来自客户端的数据。与 recv() 不同，recvfrom() 会返回数据来源的地址和端口信息。

例如：recvfrom(sock, buffer, sizeof(buffer), 0, (struct sockaddr *)&client_addr, &client_len)。

参数说明：

sockfd：套接字描述符（通常为 UDP 套接字）。

buf：指向接收数据缓冲区的指针。

len：缓冲区大小，即最多接收的字节数。

flags：接收标志，通常设为 0（常用值如 MSG_PEEK、MSG_WAITALL 等）。

src_addr：指向 struct sockaddr 结构的指针，用于存放发送方的协议地址。若不关心发送方地址，可设为 NULL。

addrlen：指向 socklen_t 类型变量的指针。调用前应将其初始化为 src_addr 缓冲区的大小；调用后，内核会将其更新为实际地址长度。

返回值：

成功时返回实际接收到的字节数（对于 UDP，可能为 0，表示收到空数据报）。

返回 0 通常表示对端已关闭连接（仅适用于面向连接的套接字，UDP 中无此含义）。

出错时返回 -1，并设置 errno 以指示错误原因。

发送数据(sendto())

使用 sendto() 函数将数据发送到指定的客户端地址和端口。UDP不需要建立连接，直接发送数据包即可。

例如：sendto(sock, message, strlen(message), 0, (struct sockaddr *)&client_addr, sizeof(client_addr))。

参数说明：

sockfd：套接字描述符（通常为 UDP 套接字）。

buf：指向要发送数据的缓冲区的指针。

len：要发送的数据长度（字节数）。

flags：发送标志，通常设为 0（常用值如 MSG_DONTROUTE、MSG_OOB 等）。

dest_addr：指向目标协议地址结构的指针（如 struct sockaddr_in 或 struct sockaddr_in6），包含接收方的 IP 地址和端口号。

addrlen：目标地址结构的大小（字节数）。

返回值：

成功时返回实际发送的字节数（对于 UDP，通常等于请求发送的长度，除非发生错误）。

出错时返回 -1，并设置 errno 以指示错误原因。

关闭套接字(close())

一旦数据传输完成，使用 close() 关闭套接字。

TCP与UDP的区别

特性 TCP UDP
连接类型 面向连接(建立连接) 无连接(不建立连接)
可靠性 提供可靠的数据传输(数据包按顺序到达) 不保证可靠性和数据顺序
速度 较慢(需要连接建立和断开) 较快(不需要建立连接)
应用场景 文件传输，Web浏览，电子邮件等 实时应用(视频，语音，在线游戏等)
数据传输方式 面向字节流(数据按流传输) 面向数据报(每个数据包独立)
流量控制 有流量控制(使用滑动窗口) 无流量控制
错误检测 有错误检测和恢复机制 仅提供简单的错误检测

粘包

原因

TCP 是流式协议，不是消息包协议。
TCP 只保证字节流可靠、有序
它不认识所谓的 “一条消息”
它只管尽量高效地把字节发过去
缓冲区攒一攒、合并一下发出去，就粘在一起了

粘包只发生在 TCP 吗？
TCP：会粘包，非常常见
UDP：不会粘包
因为 UDP 是数据报协议，发一个包就是一个包，不会合并。

解决方法

固定长度

每条消息固定 N 字节，不够补空格 / 0

分隔符结尾

每条消息末尾加特殊符号，比如 \n、|、##end##

长度头 + 数据体（最常用）

前 2/4 字节存长度，后面才是内容

滑动窗口

滑动窗口干两件事：

控制发送速度 + 保证可靠传输，是 TCP 里最核心的机制之一。

概念

滑动窗口 = 允许在没收到确认之前，连续发多个数据包，

窗口大小 = 最多能同时发多少个包。

它像一个 “窗口” 在数据序列上滑动，所以叫滑动窗口。

发送方窗口

窗口里有三种数据：
1.已发送并已确认（ACK 收到）

窗口左边，已经安全送达，可以滑走。
2.已发送但未确认

在窗口里，等待对方回复 ACK。
3.未发送，但可以马上发

在窗口右边，还没发，但允许发。

当收到 ACK 后，整个窗口向右滑动，继续发新数据。

接收方窗口

接收方会告诉发送方：

“我的缓冲区还剩多大，你最多能发多少。”

这个大小就是 接收窗口 rwnd。

发送方不能超过这个值，否则接收方处理不过来会丢包。

这就是流量控制。

滑动窗口的作用

1**.提高传输效率**

不用发一个等一个。
2.流量控制

防止发送太快把接收方冲垮。
3.配合丢包重传

哪个包没 ACK，就只重传那个，不用全部重发。

time_wait

TCP 主动关闭连接的一方，在最后会进入的一段等待状态，持续 2MSL 时间。

出现的地方

正常 TCP 断开连接是四次挥手：
A → FIN → B （A 说：我关了）
B → ACK → A （B 说：收到）
B → FIN → A （B 说：我也关了）
A → ACK → B （A 说：收到）
发最后这个 ACK 的一方，就会进入 TIME_WAIT。

为什么需要

① 防止最后一个 ACK 丢了

如果 A 发完 ACK 直接跑路：

  • 这个 ACK 在网络上丢了
  • B 没收到，就会重发 FIN
  • A 已经关闭端口，B 收到 RST 复位，会报错

TIME_WAIT 期间，A 还活着，能处理重传的 FIN，再补发 ACK。

② 让旧连接的数据包在网络中 “过期消失”

网络里可能还有延迟的旧包。

如果不等它们消失，立刻重用同一个端口，新连接可能收到旧包，导致错乱。

等待时间是 2MSL：

  • MSL：报文最大生存时间（一般 30s~2min）
  • 2MSL：一来一回，确保网络里的包都死干净

状态时长

Linux 下默认：

TIME_WAIT = 60s × 2 = 120 秒

常见问题

大量 TIME_WAIT 会怎样？

  • 占用端口资源
  • 高并发下可能端口耗尽，无法新建连接

怎么优化？

  • 开启 tcp_tw_reuse：允许 TIME_WAIT 端口快速重用
  • 开启 tcp_tw_recycle：快速回收（高版本 Linux 已不推荐）
  • 调小 tcp_fin_timeout

http/https请求方法和应答状态码

HTTP 请求方法（常用 8 个）

GET：获取资源（查），参数在 URL，明文可见，可缓存

POST：提交数据（增），参数在 body，不可缓存

PUT：更新资源（改），完整替换

PATCH：部分更新资源

DELETE：删除资源

HEAD：只拿响应头，不拿 body

OPTIONS：查询服务器支持哪些方法（跨域预检用）

CONNECT：建立隧道（HTTPS 代理用）

HTTP 状态码

1xx 信息类

100 Continue：继续发送

2xx 成功

200 OK：成功
201 Created：创建成功（POST/PUT）
204 No Content：成功但无返回内容

3xx 重定向

301 Moved Permanently：永久重定向
302 Found：临时重定向
304 Not Modified：资源未修改，用缓存（协商缓存）

4xx 客户端错误

400 Bad Request：请求格式错
401 Unauthorized：未认证（没登录）
403 Forbidden：禁止访问（权限不够）
404 Not Found：资源不存在
405 Method Not Allowed：方法不支持
429 Too Many Requests：请求太多限流

5xx 服务端错误

500 Internal Server Error：服务器内部错误
502 Bad Gateway：网关错误
503 Service Unavailable：服务不可用 / 超载
504 Gateway Timeout：网关超时

Http和Https的区别

HTTP：明文传输，端口 80，不安全

HTTPS：HTTP + TLS/SSL 加密，端口 443，安全

HTTPS 需要证书，握手耗时更长，性能略低
